## GOVEE H6076 — MATTER IS THE RIGHT PATH

### VLAN Dead End (Do Not Repeat)
- govee_light_local + govee2mqtt both fail cross-VLAN (UDP multicast 239.255.255.250:4001)
- Govee firmware replies on UDP 4002 back to source IP — never crosses VLAN regardless of firewall rules
- GOVEE_LAN_SCAN env var cannot be set via HA add-on config UI — requires Docker on IoT VLAN device
- Conclusion: govee2mqtt cross-VLAN is unsolvable without a host physically on IoT VLAN

### Matter Commissioning — Critical Steps
1. Power cycle the lamp first — commissioning window opens ~15 min after power-on
2. HA: Add Integration → Matter → "No, it's new" (NOT "Yes, already in use")
3. Enter static pairing code from Govee app: Device Settings → Matter
4. Aqara M3 hub MUST be online — it is the Matter bridge for IoT VLAN devices
5. Result entity: light.kitchen_govee_floor_lamp

### Why "No, it's new" Works After Power Cycle
- "Yes, already in use" requires Govee app to generate a fresh one-time sharing token
- Govee app does NOT expose "Share to another platform" button
- Power cycling resets Matter commissioning window — treat as factory fresh

## AQARA M3 HUB — DIAGNOSTIC PATTERN

### Multiple Matter Devices Unavailable Simultaneously
1. Check button.aqara_hub_m3_identify state FIRST
2. Same last_changed timestamp across all Matter devices = hub outage, not device issue
3. Power cycle hub (unplug 10s, replug) before any other troubleshooting
4. Hub LED: yellow → solid blue = back online
5. All downstream Matter devices restore automatically within ~60 seconds

## platform: template audit — notifications_system.yaml

### What we found
- Ran full grep across all YAML: `find /homeassistant -name '*.yaml' -not -path '*/.storage/*' | xargs grep -n 'platform: template'`
- Only ONE live instance: `packages/notifications_system.yaml:318`
- All other hits were in `/homeassistant/backups/` — irrelevant

### Critical distinction (researched against 2026.3 docs)
- `platform: template` in automation TRIGGERS is NOT deprecated — no 2026.6 deadline
- The 2026.6 deprecation is ONLY for template ENTITIES (sensor/binary_sensor/switch defined with `platform: template`)
- These are completely different things; easy to conflate

### Fix applied
- Replaced `platform: template` + `value_template` trigger watching `device_tracker.galaxy_s26_ultra` `ap_mac` attribute
- Replaced with native `trigger: state` + `attribute: ap_mac` + `to: "70:a7:41:c6:1e:6c"` (2nd floor UZ In-Wall AP)
- Native attribute trigger: only fires on transition TO that MAC, ignores all other attribute/state changes
- Template trigger gotcha avoided: template triggers can false-fire on HA restart when condition is already true

### Pattern: attribute-based state trigger
```yaml
# OLD (works but not idiomatic, restart false-fire risk)
- platform: template
  value_template: "{{ state_attr('device_tracker.foo', 'ap_mac') == 'XX:XX' }}"

# NEW (canonical 2026.3 best practice)
- trigger: state
  entity_id: device_tracker.foo
  attribute: ap_mac
  to: "XX:XX"
```
